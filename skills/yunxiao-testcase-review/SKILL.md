# 云效测试用例目录评审

## 参数

- `<organization_id>`: 云效组织ID (从 CLAUDE.md 上下文获取)

---

## 🔍 AI 评审规则（核心）

### 6 维评审标准

| 评审维度 | 通过条件 | 权重 | 分值 |
|---------|----------|------|------|
| 📝 标题清晰度 | 简洁明确，包含功能点和场景，无歧义，不包含版本号 | 高 | 2分 |
| 📋 前置条件 | 独立、完整、可执行，不依赖隐含知识，不为空或仅HTML结构 | 高 | 2分 |
| 🧪 测试步骤 | 编号清晰、操作明确、步骤完整、可复现，每条步骤有明确操作主体 | 极高 | 2分 |
| ✅ 预期结果 | 每个步骤对应明确预期结果，可验证、可量化 | 极高 | 2分 |
| 🎯 场景覆盖 | 包含正常流程、边界值、异常场景，覆盖主要分支逻辑 | 中 | 1分 |
| ⚙️ 可维护性 | 无冗余步骤、无重复用例、结构规范、术语统一 | 中 | 1分 |

**总分 10 分制**：
- **✅ 通过（8-10分）**：所有核心维度满足要求，用例质量良好
- **⚠️ 需优化（5-7分）**：1-2 项核心项不达标，有改进空间
- **❌ 不通过（<5分）**：3 项以上核心项不达标，或步骤/预期为空

### 自动优化规则

对 **需优化** 和 **评审通过但有改进空间** 的用例，自动生成优化版本，包含：

```
optimized: {
  "preCondition": "优化后的前置条件（消除HTML结构、补全缺失信息）",
  "subject": "规范化后的用例标题（移除版本号前缀、明确场景）",
  "directory": "目录名称",
  "testSteps": [
    {"step": "优化后的步骤描述", "expected": "明确、可验证的预期结果"},
    ...
  ]
}
```

**优化原则**：
1. **前置条件**：补全缺失信息、去除冗余HTML结构、明确执行前提
2. **标题**：移除版本号前缀（如 "V2.67 "）、精简表述、明确场景
3. **步骤**：每条步骤仅包含一个操作、操作主体明确、无复合步骤
4. **预期结果**：与步骤一一对应、可量化可验证、消除模糊表述
5. **场景补全**：对步骤过少的用例（<3步）补充边界场景

## 完整工作流

### Phase 1：选择范围

**Step 1.1 — 选择测试用例库

> 测试用例库从 `config/` 目录下项目配置文件中 `test_repos` 数组中读取，数组项包含 `repo_id` 和 `title` 字段。

读取 `config/` 目录下所有 YAML 配置文件，提取所有项目的 `test_repos` 配置，合并为测试用例库列表，展示给用户选择：

以表格形式展示，只显示序号和用例库名称：

| 序号 | 用例库名称 | 所属项目 |
|------|-----------|---------|
| 1 | xxx接口用例库 | xxx项目 |
| 2 | xxx用例库 | xxx项目 |

让用户通过输入序号来选择测试用例库。

**Step 1.2 — 关键词筛选目录**

由于目录数量众多，必须先通过关键词筛选再选择：

1. 调用 `mcp__yunxiao__list_testcase_directories` 获取完整目录树
2. 提示用户输入关键词筛选目录（支持多个关键词用空格分隔）：
   ```
   请输入目录关键词进行筛选（多个关键词用空格分隔，如 "登录 支付"）：
   ```
3. 匹配规则：目录名称 **包含任意一个关键词** 即命中，不区分大小写
4. 筛选后以树状结构展示匹配到的目录，标记序号：
   ```
   匹配到 5 个目录：
   ─────────────────────────
   [1] 登录模块
     [2] 密码登录
     [3] 验证码登录
   [4] 支付模块
     [5] 微信支付
   ```
5. 让用户通过输入序号选择要评审的目录，支持：
   - 单选：`1`
   - 多选（逗号分隔）：`1,2,3`
   - 全选匹配结果：`all`

**特殊情况处理**：
- 无匹配结果时：提示用户重新输入关键词，或输入 `*` 显示全部目录
- 用户输入 `*` 时：展示完整目录树供选择

---

### Phase 2：获取待评审测试用例

**Step 2.1 — 按目录拉取用例**

对每个选中的目录，调用 `mcp__yunxiao__search_testcases` 获取该目录下所有测试用例：

```python
mcp__yunxiao__search_testcases(
  organizationId="<organization_id>",
  testRepoId="<testRepoId>",
  directoryId="<directory_id>",
  page=1,
  perPage=200
)
```

循环分页处理，直到该目录下所有用例拉取完成。

**Step 2.2 — 汇总并确认**

汇总所有选中目录下的测试用例，去重后展示：

```
待评审测试用例统计
━━━━━━━━━━━━━━━━━━━━━
选中目录：{目录名称列表，用顿号分隔}
共计测试用例：{N} 条

是否开始执行 AI 评审？（输入 y 开始，或输入数量如 "前10条"）
```

如果结果 > 20 条，强烈建议用户分批处理，先处理前 20 条验证效果。

---

### Phase 3：AI 逐条评审

在内存中构建评审数据数组，每条用例数据结构为：
```javascript
{
  id: "用例ID",
  directory: "目录名称",
  original: { /* 完整原始用例信息，来自 get_testcase */ },
  review: {
    conclusion: "pass|optimize|reject",
    score: 10,
    issues: ["问题1", "问题2"],
    suggestions: ["建议1", "建议2"],
    comment: "完整评审评论内容"
  },
  optimized: { /* AI 自动优化后的用例内容，需优化/有改进空间的用例必填 */
    preCondition: "规范化后的前置条件",
    subject: "规范化后的用例标题",
    directory: "目录名称",
    testSteps: [
      {"step": "优化后的步骤描述", "expected": "明确的预期结果"}
    ]
  } | null, /* 满分通过且无需优化的用例为 null */
  reviewedAt: "ISO时间戳"
}
```

对每一条待评审测试用例，告知用户"正在评审第 N/M 条：[用例标题]"，然后依次执行，并将结果存入上述数据结构：

**Step 3.1 — 获取用例详情**

调用 `mcp__yunxiao__get_testcase` 获取完整测试用例详情：

```python
mcp__yunxiao__get_testcase(
  organizationId="<organization_id>",
  testRepoId="<testRepoId>",
  testcaseId="<testcase_id>"
)
```

提取完整信息：
- 用例标题（subject）
- 前置条件（preCondition）
- 测试步骤（testSteps）
- 优先级、创建人、标签
- customFieldValues 中的评审相关字段

**Step 3.2 — 执行评审判断**

按照顶部「🔍 AI 评审规则」的 6 维标准进行质量评估：

- **✅ 评审通过（8-10分）**：所有核心维度满足要求，用例质量良好
  - 满分通过：`optimized = null`
  - 有改进空间的通过（8-9分）：仍生成优化版本

- **⚠️ 需优化（5-7分）**：1-2 项核心项不达标，有改进空间
  - 必须生成 `optimized` 优化版本
  - 必须调用评论接口写详细评审意见
  - 评论内容使用 `[AI 评审意见]` 开头

- **❌ 评审不通过（<5分）**：3 项以上核心项不达标，或步骤/预期为空
  - 必须生成 `optimized` 优化版本（给出重写方向）
  - 必须调用评论接口写详细驳回原因
  - 评论内容使用 `[AI 评审不通过]` 开头

**Step 3.3 — 生成优化后的用例（需优化/有改进空间的用例必填）**

对 **需优化（5-7分）** 和 **评审通过但有改进空间（8分）** 的用例，按照以下规则生成优化版本：

```javascript
optimized: {
  preCondition: "补全缺失信息、去除HTML结构、明确执行前提",
  subject: "移除版本号前缀、精简表述、明确场景",
  directory: "目录名称",
  testSteps: [
    {"step": "每条步骤仅一个操作、主体明确", "expected": "可验证、无歧义"}
  ]
}
```

**优化优先级**：
1. ✨ **前置条件**：空值 → 补全；含HTML结构 → 提取纯文本
2. ✨ **标题**：移除版本号（V2.67、V3.4等）、规范命名格式
3. ✨ **测试步骤**：合并冗余步骤、拆分复合步骤、补充操作主体
4. ✨ **预期结果**：消除模糊表述、与步骤一一对应、明确验证标准
5. ✨ **场景补全**：步骤<3条的用例补充边界/异常场景

**Step 3.4 — 生成评审评论内容**

**需优化评论模板**：
```
🤖 [AI 评审意见] 需优化
━━━━━━━━━━━━━━━━━━━━━
问题清单：
1. {问题1}
2. {问题2}
...

优化建议：
- {建议1}
- {建议2}

整体评分：{X}/10 分
```

**评审不通过评论模板**：
```
🤖 [AI 评审不通过]
━━━━━━━━━━━━━━━━━━━━━
严重问题：
1. {问题1}
2. {问题2}
...

修改方向：
- {建议1}
- {建议2}

请修改后重新提交评审
整体评分：{X}/10 分
```

**Step 3.5 — 回写评审结果到云效**

**发布评审评论（必须执行）**：

使用 `mcp__yunxiao__create_work_item_comment` 的变体，或如果测试用例有专门的评论接口则使用。如果没有专用评论接口，则使用自定义字段记录评审结论。

> 注意：如果没有测试用例专用的评论接口，则将评审意见写入测试用例的备注或自定义字段中。

**更新评审结果状态（如有支持的字段）**：

先调用 `mcp__yunxiao__get_testcase_field_config` 查看是否有"评审状态"、"评审结论"等自定义字段，如果有则更新：

```python
# 先获取字段配置，找到评审相关字段
mcp__yunxiao__get_testcase_field_config(
  organizationId="<organization_id>",
  testRepoId="<testRepoId>"
)
```

如果找到相应字段则更新，找不到则跳过字段更新，不要中断流程。

**重要**：如果工具调用失败，不中断流程，在最后汇总报告中标注失败的用例。

### Phase 4：保存评审结果并汇总报告

**Step 4.1 — 保存评审结果到 JSON 文件**

- 文件名示例：`回归测试-2026-04-25-14:30.json`
- 如目录不存在，自动创建 `report/{项目名}/testcase` 目录

JSON 文件结构：

## 注意事项

- **空内容处理**：测试步骤为空、预期结果为空的用例，直接标记"评审不通过"并注明原因
- **目录递归**：默认递归处理子目录下所有用例
- **评审去重**：如果用例已有 AI 评审记录，标注"(第2次评审)"并覆盖更新
- **限速保护**：每次更新间隔 1 秒，避免触发 API 限流
- **容错处理**：任何工具调用失败都不中断整体流程，在最终报告中汇总失败列表
- **评分标准**：见顶部「🔍 AI 评审规则」，总分 10 分制，8-10 分为通过，5-7 分为需优化，5 分以下为不通过
- **优化用例生成**：需优化用例必须生成 `optimized` 字段，评审通过但有改进空间的用例也可生成优化版本
- **结果持久化**：所有用例评审数据必须保存到 report/{项目名}/testcase 目录下，不可跳过 JSON 文件保存步骤
- **数据完整性**：JSON 文件必须包含完整的原始用例信息和评审结果，用于后续创建或导入用例
