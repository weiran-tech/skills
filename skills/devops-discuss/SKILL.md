# DevOps Discuss — 需求变更讨论

## 目录约定

> 修改顶级目录时只改这里，全文及 reference 文件中的路径同步全局替换即可。

- **{讨论根目录}** = `docs/discuss/` — 讨论记录的存放根目录

通过多轮结构化讨论，帮助用户厘清需求变更的目标、现状、影响和方案，最终产出一份完整的变更讨论文档。

**目标读者**：开发、产品经理
**核心原则**：先理解再建议，多轮讨论逐步收敛，结论必须可落地

---

## 项目类型检测（启动时执行一次）

检测当前项目特征，设定后续使用的变量：

| 检测条件 | 项目类型 | 组织单元 | 文档路径 | 额外规则 |
|---------|---------|---------|---------|---------|
| 存在 `composer.json` | PHP 单体 | 模块 | `docs/{module}/` + `docs/cross-module.md` | 跨模块 `use` 引用追踪 |
| 存在 `pom.xml` | Java 单体 | 模块 | `docs/`（`overview.md`、`business.md`、`contracts.md`、`flows.md`） | — |
| 存在 `package.json` + `src/` | 前端/Node | 模块 | `docs/` | — |
| 其他 | 通用 | 组件 | `docs/` | — |

检测后确定三个变量，后续全文使用：
- **{组织单元}**：模块 / 服务 / 组件
- **{文档路径}**：架构文档的查找位置
- **{保存路径}**：`docs/discuss/{域或组织单元名}/{变更主题}/discussion.md`

---

## 讨论模式

在信息收集阶段，根据用户意图判断本次讨论的模式：

- **完整讨论**：需求变更涉及方案选择，走完全部四个阶段（信息收集 → 分析推演 → 方案收敛 → 保存记录）
- **影响分析**：用户只需要了解某个变更会影响什么，不需要出具方案。跳过第三阶段（方案收敛），在分析推演完成后直接进入保存记录

判断依据：如果用户的表述侧重"这个改动会影响什么"、"帮我列一下影响点"，倾向于影响分析模式。如果用户的表述侧重"怎么改比较好"、"有什么方案"，倾向于完整讨论模式。不确定时直接询问用户。

---

## 执行

确定项目类型和讨论模式后，**读取 `references/discuss-flow.md`** 执行四阶段讨论流程。

---

## 讨论规则

1. **按阶段推进**：跳过信息收集直接给方案，往往会遗漏关键约束导致返工。先问清楚再分析，分析完再给方案。
2. **依据文档和代码说话**：影响分析的可信度取决于是否基于真实架构。从 {文档路径} 下的文档或实际代码中提取事实，而非凭经验猜测——猜错一个事件级联就可能导致方案方向性错误。
3. **标注不确定性**：文档和代码中找不到的信息标注"待确认"。标注缺失比编造细节更有价值，因为它能引导用户补充真正需要的信息。
4. **尊重历史结论**：检查 `docs/discuss/` 下同业务域的历史讨论。之前的讨论可能已经排除了某些方案或确立了设计原则，推翻它需要说明新的理由。
5. **保持讨论节奏**：每次输出后等待用户反馈。需求讨论是双向的，用户在每个阶段都可能提供改变方向的关键信息。
6. **用业务语言**：这份文档的读者是产品经理和开发，过度技术化的表述会降低讨论效率。
