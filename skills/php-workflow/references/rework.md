# rework：设计/实现缺陷返工通道

> 处理 `/workflow rework` 时读本文件。

**返工通道**：当任意阶段（通常阶段 4/5 人工审核）发现设计或实现缺陷，需要回退重做时使用。处理"局部改写补不了、必须回到设计甚至需求层"的情况。

## `/workflow rework [需求ID][#里程碑]`

1. 解析需求 ID / `#里程碑`（省略时按活动上下文/自动选中/列出选择）
2. 与用户确认**缺陷描述**与**根因层级**（实现级 / 设计级 / 需求级），写返工单 `.task/[milestones/{里程碑}/]rework/R{轮次}-{YYYY-MM-DD}.md`：缺陷描述、根因层级、初判受影响任务/模块
3. **按层级决定回退深度**：

   | 根因层级 | 含义 | 回退动作 | 重跑范围 |
   |---------|------|---------|---------|
   | 实现级 | 设计对、代码错 | 不动 design-consensus | 受影响任务回 TODO → 重 code → 重 CR |
   | **设计级** | design-consensus 错 | 在 design-consensus 末尾追加 `## 返工修订 R{N}：{原因}`，把里程碑/需求阶段退回 **2 分析与设计**，修订后走 **阶段 3 清单门重审** | 受影响任务 **+ 依赖该设计的下游任务** 级联回 TODO（复杂任务重出 plan）→ 重 code → 重 CR |
   | 需求级 | 需求/理解本身错 | 回 **阶段 1** 重新 `/devops-discuss`，再走 2→5 | 视讨论结论确定，可能整体重来 |

4. **依赖扩散（自动算 + 人工确认）**：从 dev-tasks 依赖图自动算出"依赖被改设计 / 被改任务"的**下游任务**，连同初判受影响任务一起**列给用户确认**（可增删）；确认后才把这些任务置 TODO 并标 `(返工R{N})`。**未被勾选的 DONE 任务保留，不重复劳动**
5. **回写 progress.md**：受影响任务状态→TODO（标返工轮次）；里程碑/需求阶段按层级回退；在进度表下追加「返工记录」一行（轮次 / 日期 / 层级 / 影响任务数 / 返工单路径）
6. 之后用 `/workflow next` 正常推进重跑——设计级先过阶段 3 重审，复杂任务重出 plan，再 code → CR → 收尾验收

> rework 是**显式人工触发**，不自动判定"要返工"。它与阶段 4 CR 改写的区别：CR 改写是"设计对、修小问题"的任务内循环；rework 是"设计/需求层错了、要回退重做"的跨任务级联。拿不准选哪个：只改当前任务能解决 → 用 CR 改写；牵动设计或别的任务 → 用 rework。
