# /fe-workflow release — 发布后文档收尾

PR 合并后的核心任务：**把本次功能的业务说明增量补充到项目功能文档**，逐步构建完整的功能手册。附带进度归档、mock 扫描等收尾。

**Input**: `/fe-workflow release [--feat=<id|name>] [--module=<模块名>]`

## 执行步骤

### 1. 确认发布状态

```bash
gh pr list --head "$(git branch --show-current)" --state merged
```

备选按功能名搜索。未找到已合并 PR → AskUserQuestion 确认后继续，不阻塞。

### 2. 【核心】增量更新项目功能文档

**2.1 确定文档约定（从项目自己的规范 skill 读取，绝不硬编码某项目的路径）**：

1. 按 SKILL.md「项目约定」解析顺序找到项目编码规范 skill（如 `fe-kejinshou-h5-vue`）
2. 读取其中的 **「文档维护约定」** 章节：文档根目录、结构、页面模板、命名规则、侧边栏文件、更新规则、校验命令
3. 后续写入**严格按该约定执行**

**回退**：项目无规范 skill 或未声明「文档维护约定」→ 询问用户文档放哪（建议默认 `docs/features/{module}.md` 通用结构），并提示在规范 skill 里补充该章节。

**2.2 确定所属模块**：从本功能 spec 或改动文件路径提取模块名（如 `src/views/goods/` → goods）；跨模块改动归入主要改动所在模块。

**2.3 按约定写入**：
- **结构化项目**（约定了 1:1 镜像、按页面拆分等）：新增页面 → 按项目模板新建页面文档 + 更新模块 index 页面清单 + 挂侧边栏；改已有页面 → Edit 相关字段（接口/交互/跳转/坑），不重写整篇；改子组件 → 改所属页面文档的「组件依赖」段
- **回退项目**：写 `docs/features/{module}.md`（模块概述 / 页面清单 / 功能详情：版本、入口、核心流程、关键组件、接口依赖、业务规则）

**2.4 内容来源**：spec「一、需求」→ 操作流程、业务规则；「二、设计」→ 组件、接口依赖；代码实现 → 路由、关键配置、坑。**不写代码细节**，只写业务层面的"做什么"和"怎么用"。

**2.5 索引与校验**：新页面挂进约定的侧边栏文件；项目约定了校验命令（如 `pnpm docs:build`）→ 收尾跑一次，失败则修复后再交付。

**2.6 打戳与记账**（若项目约定声明了「更新账本」）：
- **打戳**：本次新建/修改的每篇文档 frontmatter 写入 `updated: YYYY-MM-DD` 与 `commit: <短hash>`（取 merge commit，独立模式取 HEAD）
- **记账**：账本「更新记录」表最上方追加一行（日期/commit 区间/来源 `fe-workflow release {feat-name}`/文档清单/摘要）
- **不推进**账本的 `last_sync` 锚点——本命令只覆盖本功能文档，同期可能还有其它未记账变更（锚点由项目的文档时间同步工具负责）

### 3. 更新 CLAUDE.md（仅"新增模块级别"变更）

新增业务模块 → 追加到模块列表；新增公共组件/工具 → 追加对应说明；小功能扩展/Bug 修复 → 不更新。

### 4. 归档进度

progress.yaml：该功能所有步骤标 done，`shipped` 数组追加记录（格式见 `templates.md`）。

### 5. 扫描 mock 标记

`grep -rn "TODO: 替换为真实接口" {本功能 files_changed}`，有未替换 → 提醒 `/fe-workflow api-sync`。

### 6. 写入 Memory（仅非显而易见的决策，若环境支持）

从 decisions 筛选只写 feedback 类型：技术偏好、被否决的方案、项目级约定；不写可从 git/代码查到的信息。

### 7. 漏网变更检查（可选）

若安装了文档时间同步工具（如 `lc-doc-sync`）且项目声明了「更新账本」→ 按其 `--check` 模式跑一次**只读**检查（账本锚点 → HEAD 映射过时文档），零成本发现同期绕过流水线的变更（hotfix、重构、他人合并）还有哪些没进文档。只报告不修改、不推进锚点；发现漏网 → 总结中列清单并建议用户跑对应工具补齐。未安装或无账本 → 跳过。

### 8. 输出总结

文档更新情况 / 漏网变更检查结果 / mock 标记情况 / 分支清理建议（不自动执行）。

## 独立使用：补写历史功能文档

`/fe-workflow release --module=goods` — 不依赖流水线产物：跳过 PR 确认和归档，直接读取模块代码（路由、组件、接口调用）生成功能文档写入。适合逐步梳理存量未文档化的功能。

## Guardrails

- **文档结构以项目规范 skill 的「文档维护约定」为准**，不内置任何项目路径或模板
- 只写业务层面，不写代码实现细节
- 追加用 Edit，不覆盖已有内容；同名功能条目更新而非重复追加
- CLAUDE.md 只加模块级新增；mock 只提醒不替换；分支清理只提醒不执行
