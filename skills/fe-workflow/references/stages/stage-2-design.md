# Step 2 — 技术设计（design，含自审 + 冲突预检）

根据本功能 spec 的「一、需求」，在**同一文件**追加「二、设计」章节；生成后立即自审并直接修订。**不生成独立的设计文件。** lite 模式跳过本步（标记 `skipped`）。

**前置检查**：本功能 spec 的「一、需求」必须存在，否则先执行 Step 1。

## 执行步骤

### 1. 读取需求并判断必要性

读取 spec「一、需求」。若需求类型为"功能扩展/Bug修复/重构"且改动量极小，建议跳过设计直接实现（等同 lite）。

### 2. 加载项目编码规范

按 SKILL.md「项目约定」的规范 skill 解析顺序（config.yaml > 项目 CLAUDE.md > 映射表）加载项目编码规范，作为设计的技术标准——规范中的组件库、请求模式、路由模式优先级高于本文件的通用描述。

并行读取以下内容，确保设计与项目风格一致：
- 一个相似业务模块的页面文件
- 对应的 service/request 文件
- 路由配置文件

### 3. 尝试获取 API 接口

- 通过 Apifox MCP 获取相关接口（如项目已配置），记录到设计文档
- 未获取到 → 标注使用 mock 数据，并设计 mock 数据结构

### 4. 生成设计章节

在 spec 末尾追加「二、设计」（模板见 `templates.md`），并把「基本信息」的状态更新为"设计完成"。

**去重规则（硬性）**：
- 不重复需求部分已有内容——需求背景、操作流程、字段表只在「一、需求」保留，设计章节需要时用"见一、需求"引用
- 需求部分已有"影响面分析"改动文件清单时，设计章节只补充新增文件并引用需求部分，不整表重抄
- 接口信息只在"接口与 Mock"展开一次

### 5. 设计自审（生成后立即执行，不是独立步骤）

**5.1 收集参考材料**：用户附带了截图/参考文档 → Read 读取并纳入审查依据。

**5.2 按维度审查**：

| 维度 | 检查点 |
|------|--------|
| A 需求覆盖度 | 每个功能点有对应设计；改动文件清单完整；数据字段全覆盖 |
| B UI/交互一致性（有截图时） | 布局与截图一致；交互流程、操作入口完整 |
| C 技术合理性 | 组件选型、请求层、路由模式是否符合**项目编码规范 skill** 的约定 |
| D 架构快检 | 数据流清晰（props/events/emitter）；无不必要的状态提升，store 是否真的需要；组件职责单一、拆分粒度合适；与项目现有同类模块架构一致 |
| E 边界场景 | 空数据/加载/错误处理；大列表分页或虚拟滚动；搜索输入防抖 |

**5.3 自动修订**：发现问题直接修改「二、设计」，并在文档末尾「自审修订」表追加记录。

**5.4 质量门控**：无严重问题或已修订 → 通过；有 P1 级严重问题（架构方向错误、核心组件选型不当）→ **AskUserQuestion** 提示用户确认修订后的设计。

### 6. 跨功能文件冲突预检（多功能流水线必做）

将本功能设计的"改动文件清单"与同流水线其他功能的清单（设计预估 + 已回写的 files_changed）对比：
- 有重叠文件 → 在设计摘要中**明确标出重叠文件与建议执行顺序**（后跑的功能基于前者的改动编写），并记入 decisions

### 7. 合并审核（本功能第一次暂停）

展示**需求要点摘要 + 设计摘要 + 自审修订记录 + 冲突预检结果**，等待用户审核：
- 审核通过 → 置 `design_approved: true`
- `--until=design` → 流水线结束，提示"设计已就绪，可继续添加下一个需求，或晚上用 --start 批量执行"
- 否则 → `是否继续 Step 3（编码实现）？ y — 继续 / s — 跳过`
- 用户对需求理解有异议 → 修订「一、需求」并同步重跑受影响的设计章节，记入 decisions，再次展示确认

## Guardrails

- **必须先加载项目编码规范 skill**，用对应的组件库、请求模式、路由模式；跨项目组件体系不得混用
- `<script setup lang="ts">` + Composition API（各前端项目通用）
- 设计必须参考项目中相似模块的实际代码，不能凭空设计
- 设计必须覆盖：路由、页面、Service/Request、权限（如适用）四个维度
- 设计自审是本步骤的一部分，不单独生成审查文件
- **严格遵守去重规则**：同一信息全文档只出现一次
