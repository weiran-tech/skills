# Step 1 — 需求分析（requirement）

将用户输入（文字/截图/文档）转化为结构化需求，写入本功能 spec 文件（`spec-{n}-{feat-name}.md`）的「基本信息」+「一、需求」部分。

## 执行步骤

### 1. 获取需求信息

如果用户没有提供足够的需求描述，使用 **AskUserQuestion tool** 询问：
> "请描述你想要实现的功能，可以提供文字说明、截图路径或相关文档。"

**feat-name 命名规则**：用户在需求描述中显式给出功能名则使用之；否则从描述中提取 kebab-case 短名（如"添加商品管理" → `goods-manage`）。需求分析完成后若临时编号（`spec-{n}.md`），据此补名并重命名文件、同步 progress.yaml。

### 2. 识别需求类型

| 类型 | 判断依据 | 分析重点 |
|------|---------|---------|
| **新功能** | 需要新页面、新路由 | 页面清单、数据字段、完整流程 |
| **功能扩展** | 在现有模块上增加分支/选项 | 影响面分析、改动文件清单 |
| **Bug 修复** | 修复现有功能的异常 | 问题复现、根因定位、修复方案 |
| **重构优化** | 不改变功能，改善代码/性能 | 改动范围、风险评估 |

### 3. 分析需求上下文

- 用户提供了截图 → 必须用 Read tool 读取并分析
- 分析用户提供的文档内容
- **检测项目类型**（package.json + src/ 目录结构），确定后续用哪套术语和组件体系（见 SKILL.md「项目约定」的规范 skill 映射表）
- 了解项目已有的相关模块（按项目类型检查对应目录）
- 检查是否有 Apifox 接口文档可用（项目配置了 Apifox MCP 时，工具名从当前可用 MCP 工具列表选取）

### 4. 影响面分析（功能扩展/Bug修复/重构 类型必须执行）

防止遗漏需要同步改动的文件：

1. **找到改动入口**：确定需要修改的核心代码点（枚举、配置、组件）
2. **搜索所有引用**：用 Grep 搜索核心代码点的全部引用位置
3. **逐个判断**：对每个引用点，分析其逻辑是否依赖本次改动
4. **输出影响面清单**：列出所有需要改动的文件，标注改动类型和原因

> 示例：扩展支付方式时，不仅要改支付枚举和分发逻辑，还要 grep 同类枚举的引用，发现支付方式列表构建逻辑所在的组件也需要同步添加。

### 5. 生成需求部分

按需求类型选用 `templates.md` 的模板 A（新功能）或模板 B（扩展/修复/重构），只写「基本信息」和「一、需求」，「二、设计」由 Step 2 追加。

### 6. 模式建议与接续

- 根据需求类型建议 full 或 lite 模式，记录到该功能的 `mode`
- **full 模式：不暂停**，直接进入 Step 2（需求审核合并到设计完成后统一进行）
- **lite 模式：暂停**（design 被跳过，需求部分是唯一文档）：`是否继续 Step 3（编码实现）？ y — 继续 / s — 跳过`

## Guardrails

- 用户提供的截图必须实际读取分析，不能只看文字描述
- 尽量从 Apifox MCP 获取接口信息，获取不到时标注"待定 - 使用 mock 数据"
- **功能扩展型需求必须执行影响面分析**，grep 所有相关引用，不能只看核心入口文件
- "改动文件清单"必须逐文件标注改动原因，不能写"无需改动"然后遗漏
- 按项目类型使用对应术语（H5 项目不要写 KrCard/KrTable 等后台组件名，反之亦然）
- **需求部分不展开接口细节、不单列页面清单**——这些在「二、设计」统一呈现，同一信息只写一处
