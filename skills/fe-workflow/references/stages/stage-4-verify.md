# Step 4 — 代码验证（verify：Lint + Test + QA/CR）

对本功能代码执行三阶段验证：静态检查 → 单元测试 → QA+CR（独立子代理），结果写入 `verify-report.md` 本功能章节。

**两种调用方式**：

- **流水线内**（Step 4）：前置检查本功能 implement 为 done；**分析范围以本功能的 files_changed 为准**，不把同分支其他功能的未提交改动算进本功能的报告
- **独立调用**（`/fe-workflow verify`，不依赖流水线）：跳过前置检查；分析范围兜底 `git diff --name-only master`（含未提交改动）；无 spec 时跳过"需求覆盖度"检查、其余照常；结果直接输出到对话（带 `--feat` 定位到流水线功能时才写汇总报告章节）

## 准备

- 读取本功能 spec（需求+设计摘要）+ files_changed 列出的源代码文件
- 按 SKILL.md「项目约定」的规范 skill 解析顺序加载项目编码规范，作为检查的技术标准

## 阶段一：静态检查（Lint）

### 1. 首次检查

```bash
npx eslint {files_changed} 2>&1
```

### 2. 区分新增错误 vs 原有错误

对比变更文件在 master 上是否已有相同错误：**原有错误**记录但不阻塞；**新增错误**（本次改动引入）必须处理。

### 3. 自动修复

```bash
npx eslint {files_changed} --fix 2>&1
```

修复后重新检查确认。

### 4. 安全自动修复白名单（--fix 之外，仅限以下类型）

| 可自动修复（Edit 直接改） | 禁止自动修复（只报告） |
|--------------------------|----------------------|
| 未使用的 import / 未使用且无副作用的变量声明 | 条件判断修改（`==` → `===`） |
| 重复的 import（合并为一行） | 变量重命名 |
| 多余的空行/空格 | 类型标注修改 |
| console.log 遗留（仅新增代码中的） | 函数参数增删 |
| | 任何涉及条件分支/数据流的改动 |

> **铁律：有任何不确定是否影响逻辑的，一律不自动修复，只在报告中标注。**

## 阶段二：单元测试（Test）

### 测试范围规则

**只测新增的核心函数，不测页面组件本身。**
- 核心函数：新增的**纯函数**（label 映射、数据格式化、枚举转换）、**工具函数**（计算、校验、数据变换）、**常量文件**（结构完整性）
- 不测：Vue 页面组件渲染/交互、Service 层 API 封装、路由配置、权限配置
- **跳过条件**：无新增核心函数 → 跳过并在报告标注

### 执行（有核心函数时）

1. 检测框架：`npx vitest --version`（vitest 与 jest 都不可用 → 跳过并标注）
2. 生成测试：每个 it 只测一个行为；断言具体值（不写 `toBeTruthy()` 之类模糊断言）；覆盖正常/边界/异常值
3. 执行：`npx vitest run {测试文件} --reporter=verbose`；失败用例自动诊断修复，最多 3 轮（优先修测试代码，不随意改源码）
4. 回归：项目有已有测试 → 跑全量确认无回归

## 阶段三：QA + CR（独立子代理）

**必须派独立子代理执行**（全新上下文，只读权限）——编码与审查不共享上下文，禁止同上下文自审。

### 子代理任务书（spawn 时提供）

输入：本功能 spec 全文、files_changed 文件列表、`git diff master -- {files_changed}` 的 diff。任务：

1. **需求覆盖度**：逐项对照 spec「一、需求」，检查功能点实现、数据字段完整、操作流程一致
2. **质量维度分析**：逻辑完整性（分支遗漏/异步错误处理/loading）、边界情况（表单校验/分页参数/危险操作二次确认）、安全性（XSS·v-html/权限控制/敏感数据脱敏）、性能（重复请求/大列表/重渲染）、用户体验（操作反馈/防重复点击/错误信息友好性）、风格与项目规范一致性
3. **变更影响分析**：Bug 模式检测；删除/重命名 grep 验证无残留引用；行为变更检测（条件分支、语义漂移、接口签名传播）
4. **输出问题清单**，按级别分类：
   - **A 清理类**：不影响逻辑的清理（未使用 import、TODO 格式、遗留 console.log、空代码块、注释 typo）
   - **B 建议项**：质量改进但不阻塞（命名语义化、可简化逻辑、缺边界处理、魔法数字）
   - **C 阻塞项**：正确性/安全性问题（逻辑错误、接口参数不匹配、安全风险、破坏性影响未同步）
5. 输出各维度评分（/10）+ 需求覆盖率

**子代理只产清单不改码。** 返回后主流程必须：读取清单 → 回写 progress.yaml → 输出摘要与下一步提示（不静默结束）。

### 门控与修复（主流程执行）

- **零 C 类问题** → 自动通过：A 类由主流程按清单修复（修复后重跑 ESLint 确认无新错误），B 类写入报告即可，明确告知"verify 通过，自动继续"
- **有 C 类问题** → **暂停**，逐条展示等用户确认：确认修复 → 主流程修复后复验；用户判定误报 → 标注忽略原因。全部处理完才继续
- 若 verify-report 已记录某问题，不重复报告

## 生成报告与门控

写入 `verify-report.md` 本功能章节（模板见 `templates.md`，只覆盖本功能章节）。

- **通过条件**：新增代码 Lint 0 error + 测试通过或合理跳过 + QA ≥ 8 分且无未处理 C 类问题
- **阻塞条件**：无法自动修复的新增 ESLint error / 测试 3 轮修复仍失败 / C 类问题未处理 / QA < 6（需回到实现阶段）
- 通过后：`是否继续 Step 5（E2E 测试）？ y — 继续 / s — 跳过`

## Guardrails

- **ESLint 与测试必须实际执行**（`npx eslint` / `npx vitest run`），不能仅靠人工阅读或只写不跑
- **安全自动修复严格遵守白名单**；绝不修改条件判断、数据流、函数签名、业务逻辑；修复前后必须重新 ESLint 验证
- 原有代码错误不计入门控判定
- **QA+CR 必须派独立子代理**，子代理只读不写；主流程只修复 A 类和已确认的 C 类
- QA 必须对照需求文档逐项检查，不凭印象；检查标准与项目类型匹配（H5 不检查后台权限体系，反之亦然）
- 评分标准：8+ 通过，6-7 修复后重新验证，<6 回到实现阶段
