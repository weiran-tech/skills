# templates — progress.yaml / spec / 报告模板

## progress.yaml（状态权威源）

```yaml
pipeline: {pipeline-name}
branch: feature/xxx          # 创建时的分支名，入口扫描与看板按此归属
created: YYYY-MM-DD HH:mm
updated: YYYY-MM-DD HH:mm
features:
  - id: 1
    name: coupon-list
    spec: spec-1-coupon-list.md
    mode: full               # full|lite，按功能独立
    design_approved: false   # 设计经人工审核通过后置 true（--start 只执行 true 的功能）
    steps:
      requirement: { status: pending }
      design: { status: pending, depends: [requirement] }
      implement: { status: pending, depends: [design] }
      verify: { status: pending, depends: [implement] }
      e2e: { status: pending, depends: [implement] }
    files_changed: []        # implement 完成后回写实际改动文件列表
  - id: 2
    name: coupon-export
    spec: spec-2-coupon-export.md
decisions: []                # 每条记录必须带 feature 字段
shipped: []                  # release 子命令按功能追加
```

- **status 枚举**：`pending` | `in-progress` | `done` | `skipped` | `blocked`
- **decisions 格式**：

```yaml
decisions:
  - feature: coupon-export
    step: design
    time: "YYYY-MM-DD HH:mm"
    content: "与 #1 重叠改动 CouponList.vue，约定 #1 先实现，#2 基于其改动编写"
```

- **shipped 格式**（release 子命令写入，按功能一条）：

```yaml
shipped:
  - feature: coupon-list
    date: "YYYY-MM-DD HH:mm"
    pr_url: "{PR URL}"
    branch: "{分支名}"
```

## spec 文档模板

文件名 `spec-{n}-{feat-name}.md`。requirement 写「基本信息 + 一、需求」，design 追加「二、设计 + 自审修订」。

### 模板 A：新功能（需要新页面/新路由）

```markdown
# {功能名称} 需求与设计文档

## 基本信息
- **功能名称**: {中文名}
- **功能标识**: {feat-name}
- **需求类型**: 新功能
- **创建日期**: {YYYY-MM-DD}
- **优先级**: P0/P1/P2
- **状态**: 待设计

## 一、需求

### 需求背景
{为什么要做这个功能，解决什么问题}

### 功能描述
#### 核心功能
- {功能点1}

#### 用户操作流程
1. {步骤1}

### 数据需求
#### 列表字段
| 字段名 | 类型 | 说明 |
|--------|------|------|

#### 搜索/筛选条件
| 条件 | 类型 | 说明 |
|------|------|------|

#### 表单字段（新增/编辑）
| 字段名 | 类型 | 必填 | 校验规则 | 说明 |
|--------|------|------|----------|------|

### 接口需求
{一句话概括：已有 Apifox 接口 / 待定 - 使用 mock 数据。详细接口在「二、设计」的"接口与 Mock"章节展开，此处不重复}

### 非功能需求
- 性能要求 / 兼容性: {如有}

### 参考资料
- {截图/文档/原型链接}
```

> **去重说明**：需求部分不单列"页面清单"——页面及其路由、文件在「二、设计」的"改动文件清单"中一并呈现，避免两处重复维护。

### 模板 B：功能扩展 / Bug修复 / 重构优化

```markdown
# {功能名称} 需求与设计文档

## 基本信息
{同模板 A，需求类型改为：功能扩展 / Bug修复 / 重构优化}

## 一、需求

### 需求背景
{为什么要做这个改动，解决什么问题}

### 功能描述
#### 改动内容
- {改动点1}

#### 用户操作流程（改动后）
1. {步骤1}

### 影响面分析
#### 改动文件清单
| 文件 | 改动类型 | 改动说明 | 原因 |
|------|---------|---------|------|

#### 改动范围评估
- **改动量**: 极小/小/中/大
- **风险等级**: 低/中/高
- **核心改动**: {最关键的 N 个文件}
- **联动改动**: {因核心改动需要同步修改的文件}

### 接口需求
{"无接口变更"，或一句话概括}

### 非功能需求 / 参考资料
{如有；代码位置引用格式 file_path:line_number}
```

## verify-report.md（汇总报告，按功能分章节）

每条流水线一份，每个功能一个 `## #{id} {feat-name}` 章节，更新时只覆盖本功能章节：

```markdown
# {pipeline-name} 代码验证报告

## #{id} {feat-name}
- 验证日期: {YYYY-MM-DD}　项目类型: {类型}　分析范围: 本功能 files_changed（{N} 个文件）

### 一、静态检查（Lint）
- 新增代码错误：{N} 个（--fix 修复 {M} 个）；原有代码错误：{K} 个（不在本次改动范围）

#### 安全自动修复（已执行）
| 文件 | 问题 | 修复方式 |
|------|------|---------|

#### 需手动处理（涉及逻辑）
| 文件 | 问题 | 建议 |
|------|------|------|

### 二、单元测试（Test）
- 状态：{执行 / 跳过（无新增核心函数）}
- 新增用例 {N} 个，通过 {N}；回归测试：{通过/无已有测试}

| 文件 | 用例名称 | 状态 |
|------|---------|------|

### 三、QA + CR（独立子代理审查）

#### 需求覆盖度
| 功能点 | 状态 | 备注 |
|--------|------|------|
覆盖率: {百分比}

#### 问题清单
| # | 级别 | 问题 | 文件:行号 | 处理 |
|---|------|------|----------|------|
{级别：A 清理类（已自动修复）/ B 建议项（不阻塞）/ C 阻塞项（需确认后修复）}

#### 维度评分（各 /10）
| 逻辑完整性 | 边界处理 | 安全性 | 性能 | 用户体验 | 变更影响 | 风格一致性 | 项目规范 |
|-----------|---------|--------|------|---------|---------|-----------|---------|

### 总评
- QA 评分: {0-10}/10；Lint: {状态}；测试: {状态}；C 类阻塞项: {N} 个
- 状态: {通过 / 有 N 个问题待确认}
- 建议 commit message: `feat/fix(scope): description`
```

## e2e-report.md（汇总报告，按功能分章节）

```markdown
# {pipeline-name} E2E 测试报告

## #{id} {feat-name}
- 测试日期: {YYYY-MM-DD}　框架: Playwright　设备模拟: {如 Pixel 5}
- 驱动模式: {A 需求+设计驱动 / B 现有文档驱动 / C 测试用例驱动}
- 用例来源: {spec 文件 / 文档路径 / 云效用例库+目录}

### 用例分流
| 标签 | 数量 | 说明 |
|------|------|------|
| ✅ 可 E2E | {N} | 已生成用例 |
| 🔒 需登录 | {N} | {已生成 / skipped：缺测试 token} |
| 🧩 需 mock | {N} | JSBridge / 原生能力，未生成 |
| ⬜ 不可 E2E | {N} | 白盒断言，建议单测 |

### 测试概览
- 用例 {N} 个，通过 {N}，失败 {N}（已修复 {M}）；截图 {N} 张；耗时 {N}s

### 测试用例清单
| 用例名称 | 状态 | 耗时 | 截图 |
|---------|------|------|------|

### 失败用例分析 / 发现的功能 Bug（如有）
| 用例 | 失败原因 | 类型（测试问题/功能bug） | 处理 |
|------|---------|------------------------|------|
```

```markdown
# 批量执行报告 — {YYYY-MM-DD}

## 执行概览
| # | 功能 | implement | verify | e2e | 状态 |
|---|------|-----------|--------|-----|------|
| 1 | coupon-list | ✅ | ✅ 8.6/10 | ✅ 4/4 | 完成 |
| 3 | coupon-stats | ❌ | — | — | blocked：{原因} |

## 逐功能 commit 建议（按功能分组，依次提交）
### #1 coupon-list
files: {files_changed 列表}
commit: feat(coupon): 新增优惠券列表
```

- blocked 功能列明原因和建议的人工处理方式；本报告是早上人工按功能逐个 commit 的依据。
