# business.md 生成指南

business.md 记录的是**代码读不出来的内容**：规则背后的原因、状态流转的触发条件、决策的业务依据。如果某件事直接看代码就能明白，就不需要在这里重复。

## 扫描重点

- **状态机**：含 Status、State 的枚举，以及状态变更的触发条件
- **业务规则**：Service 层中的条件判断、限制条件、幂等处理

## 输出模板

```markdown
# 业务逻辑

## {核心业务模块1，如"通知投递"、"自动化流程"、"订单处理"}


### 状态机
{如果有状态流转，用箭头图描述}

PENDING → PROCESSING → SUCCESS
                     ↘ FAILED → RETRY（最多 N 次）→ DEAD

状态说明：
- FAILED → RETRY 的条件：{可重试的错误类型}
- 不可重试的情况：{直接终止的错误类型}

### 关键业务规则
- {规则描述}：{为什么有这个规则，背景是什么}

## {核心业务模块2}
...
```

## 职责边界

business.md 只回答"为什么这么处理，规则是什么"。

**禁止出现**：
- 按步骤编号的执行流程、事件级联链路 → 改为引用："完整执行链路见 flows.md"

**跨文档引用格式**：
- 提到流程时："议价超时的完整执行链路见 flows.md"
- 禁止为了"方便"而把其他文档的内容复制过来

**"待确认"内容处理**：
- 禁止将待确认内容混入正文描述中
- 所有待确认事项统一放在文末 `## 待确认` 段落
- 格式：`- {待确认内容}（发现位置：{类名或方法名}）`

## 写作原则

- 每条规则尽量解释"为什么"，不只是"是什么"
- 状态机图优先用 ASCII 箭头，清晰直观
- 不要把代码逻辑直接翻译成文字，要提炼成业务语言
