# docs/{module}/business.md 生成指南

business.md 记录的是**代码读不出来的内容**：规则背后的原因、状态流转的触发条件、路由决策的业务依据。如果某件事直接看代码就能明白，就不需要在这里重复。

## 扫描重点

- **Action 层逻辑**：Action 类中的条件判断、业务校验、分支处理——理解"什么情况走哪条路"
- **状态枚举/常量**：模型中的状态字段、Classes 中的常量定义，以及状态变更的触发条件
- **业务规则**：Action 层中的限制条件、幂等处理、金额计算、权重算法
- **定时任务/命令**：Commands 中的 Artisan 命令、Jobs 中的定时调度逻辑
- **中间件逻辑**：Middlewares 中的请求拦截规则

## 职责边界

business.md 只回答"为什么这么处理，规则是什么"。

**禁止出现**：
- 具体的事件类名（XxxEvent）、监听器类名（XxxListener）→ 改为引用："相关事件见 contracts.md"
- 按步骤编号的执行流程、事件级联链路 → 改为引用："完整执行链路见 flows.md"
- 具体的 API 路由路径 → 属于 contracts.md

**跨文档引用格式**：
- 提到事件时："商品回收通过后触发通知，相关事件见 contracts.md"
- 提到流程时："议价超时的完整执行链路见 flows.md"
- 禁止为了"方便"而把其他文档的内容复制过来

**"待确认"内容处理**：
- 禁止将待确认内容混入正文描述中
- 所有待确认事项统一放在文末 `## 待确认` 段落
- 格式：`- {待确认内容}（发现位置：{类名或方法名}）`

## 写作原则

- 每条规则尽量解释"为什么"，不只是"是什么"
- 状态机图优先用 ASCII 箭头，清晰直观
- 不要把代码逻辑直接翻译成文字，要提炼成业务语言
- Action 类是业务逻辑的核心，重点分析其中的条件分支和校验规则
