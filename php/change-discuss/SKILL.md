# Change Discuss — 需求变更讨论

通过多轮结构化讨论，帮助用户厘清需求变更的目标、现状、影响和方案，最终产出一份完整的变更讨论文档。

**目标读者**：开发、产品经理
**核心原则**：先理解再建议，多轮讨论逐步收敛，结论必须可落地

---

## 讨论模式

在信息收集阶段，根据用户意图判断本次讨论的模式：

- **完整讨论**：需求变更涉及方案选择，走完全部四个阶段（信息收集 → 分析推演 → 方案收敛 → 保存记录）
- **影响分析**：用户只需要了解某个变更会影响什么，不需要出具方案。跳过第三阶段（方案收敛），在分析推演完成后直接进入保存记录

判断依据：如果用户的表述侧重"这个改动会影响什么"、"帮我列一下影响点"、"评估一下影响范围"，倾向于影响分析模式。如果用户的表述侧重"怎么改比较好"、"有什么方案"、"讨论一下怎么做"，倾向于完整讨论模式。不确定时直接询问用户。

---

## 讨论流程

完整流程分为四个阶段，影响分析模式跳过第三阶段。每个阶段通过与用户的多轮对话完成，不要在信息不充分时直接给结论。

---

### 第一阶段：信息收集

目标：搞清楚用户要改什么、为什么要改。

**主动询问以下信息**（用户未提供的要追问）：

1. **变更主题**：一句话概括这次要改什么
2. **业务背景**：为什么需要这个变更？是谁提出的？解决什么问题？
3. **涉及的模块**：属于哪个模块（从 `composer.json` 的 `autoload.psr-4` 或 `modules/` 目录获取模块清单）
4. **期望效果**：变更完成后，用户/系统的行为应该是怎样的

同时，**主动查阅项目中的相关文档**来补充上下文：
- `docs/workflow/{module}/` 下相关模块的 `overview.md`、`business.md`、`flows.md`、`contracts.md`
- `docs/workflow/cross-module.md` 跨模块依赖关系
- `discuss/` 下是否有同业务域的历史讨论，避免重复讨论已有结论

**如果 docs/workflow/ 目录不存在**，提醒用户先运行 `php-analyzer` 生成模块文档。讨论质量高度依赖文档，缺乏文档时的分析准确度无法保证。

将收集到的信息整理后，向用户确认理解是否正确，再进入下一阶段。

---

### 第二阶段：分析推演

目标：基于现有架构文档或代码，分析变更的影响面。

依次产出以下内容，每部分产出后与用户确认、补充：

#### 2.1 需求目标

用 2-3 句话清晰描述：
- 这次变更要达成什么业务目标
- 变更的边界是什么（做什么、不做什么）

#### 2.2 当前流程

描述当前系统在这个业务场景下的处理流程：
- 涉及哪些模块、哪些 Action
- 关键步骤的先后顺序
- 核心状态变化和事件触发链（Event → Listener → Action → Job）

**数据来源**：优先从 `docs/workflow/{module}/` 下的文档中提取，回退到直接阅读代码。不要凭空编造。如果文档和代码中都找不到相关信息，明确标注"待确认"。

#### 2.3 改动影响分析

逐一分析变更对以下维度的影响：

- **直接影响的模块**：哪些模块需要改代码，改什么（Action/Model/Event/Route）
- **间接影响的模块**：哪些模块虽然不改代码，但行为会受影响（如引用了被改模块的 Model 或 Action）
- **级联副作用**：Event → Listener 级联链、Job 异步任务、跨模块 Action 调用会受到什么影响
- **数据一致性**：是否存在短时不一致、最终一致性风险
- **向前兼容**：存量数据怎么处理、新老逻辑如何过渡
- **性能风险**：是否涉及批量操作、队列峰值、大表扫描等

每个影响点要具体到：**改了什么 → 影响哪个模块/流程 → 可能导致什么问题**。

---

### 第三阶段：方案收敛

目标：给出推荐的改动方向，可以有多个方案对比。

#### 3.1 推荐改动方向

至少给出 1 个推荐方案，复杂场景给出 2-3 个方案对比。每个方案包括：

- **方案名称**：一句话概括
- **核心思路**：怎么改、改哪里（具体到模块和类）
- **优点**
- **缺点**
- **实施复杂度**：低 / 中 / 高
- **推荐理由或不推荐理由**

#### 3.2 收敛结论

与用户讨论后，明确：
- 最终倾向哪个方案（或组合方案）
- 还有哪些待确认的点
- 是否需要进一步的技术调研或评审

**这个阶段允许多轮讨论**。用户可能会提出新的约束、质疑某个方案、要求深入某个细节。耐心回应，直到用户明确表示讨论可以收尾。

---

### 第四阶段：保存记录

当用户确认讨论完成后（用户明确说"可以保存了"、"结束讨论"、"保存"等），执行以下操作：

#### 4.1 确定保存路径

- 根据模块确定子目录：`change-discuss/{模块名}/`
- 文档名使用变更主题：`change-discuss/{模块名}/{变更主题}.md`
- 如果涉及多个模块，放在主模块目录下，文档中标注涉及的其他模块
- 如果子目录不存在，自动创建

#### 4.2 生成文档

读取 `references/templates.md`，根据讨论模式选择对应的文档模板和输出确认格式。

## 讨论示例

**示例 1：信息收集阶段的交互**

用户：某个功能需要增加一个自动过期机制

助手回应要点：
- 确认变更主题：XX功能自动过期
- 追问业务背景：是因为长期未处理导致积压？过期后的预期行为是什么？
- 主动查阅涉及模块的 `docs/workflow/{module}/business.md` 和 `docs/workflow/{module}/contracts.md`
- 检查 `discuss/{module}/` 下是否有相关历史讨论
- 整理理解后向用户确认理解是否正确

**示例 2：影响分析中的跨模块追踪**

助手分析时：
- 从 `docs/workflow/{module}/contracts.md` 中找到相关 Event 的所有 Listener
- 从 `docs/workflow/cross-module.md` 中确认跨模块引用关系
- 追踪 Listener 中是否调用了其他模块的 Action（跨模块影响）
- 追踪是否有 Job 被 dispatch（异步影响）
- 对文档中标注为"待确认"的级联关系，明确提出需要用户确认

## 讨论规则

1. **按阶段推进**：跳过信息收集直接给方案，往往会遗漏关键约束导致返工。先问清楚再分析，分析完再给方案。
2. **依据文档和代码说话**：影响分析的可信度取决于是否基于真实架构。从 `docs/workflow/` 下的文档或实际代码中提取事实，而非凭经验猜测——猜错一个事件级联就可能导致方案方向性错误。
3. **标注不确定性**：文档和代码中找不到的信息标注"待确认"。标注缺失比编造细节更有价值，因为它能引导用户补充真正需要的信息。
4. **尊重历史结论**：检查 `discuss/` 下同模块的历史讨论。之前的讨论可能已经排除了某些方案或确立了设计原则，推翻它需要说明新的理由。
5. **保持讨论节奏**：每次输出后等待用户反馈。需求讨论是双向的，用户在每个阶段都可能提供改变方向的关键信息。
6. **用业务语言**：这份文档的读者是产品经理和开发，过度技术化的表述会降低讨论效率，让非技术参与者无法有效参与决策。
7. **跨模块影响必须追踪**：本项目是伪多模块架构，模块间通过 `use` 直接引用耦合。任何涉及 Model 字段变更、Event 参数变更、Action 公开方法签名变更的改动，都必须 grep 其他模块的引用来确认影响范围。
